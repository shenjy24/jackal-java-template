# 联网部署文档

本文档说明生产环境可访问公网时，如何通过 `deploy` 目录拉取代码、构建镜像、启动服务，以及回滚到本机历史镜像。

## 目录结构

生产服务器上，`deploy` 目录与业务代码目录同级：

```text
部署根目录/
├── deploy/
│   ├── Dockerfile
│   ├── docker-compose.yml
│   ├── deploy.env
│   ├── start.sh
│   └── rollback.sh
└── java-template/
    └── 业务代码仓库
```

脚本通过 `REPO_NAME` 定位业务代码目录，默认路径为：

```text
deploy/../java-template
```

`docker-compose.yml` 位于 `deploy` 目录，构建上下文指向业务代码目录，Dockerfile 使用 `deploy/Dockerfile`。

## 前置条件

- 服务器已安装 Git、Docker、Docker Compose Plugin。
- 服务器可以访问代码仓库和基础镜像仓库。
- 当前用户可以执行 Docker 命令；如未加入 `docker` 组，请使用 `sudo bash` 执行脚本。
- 生产配置 `src/main/resources/application-prod.yml` 已按真实环境配置完成。
- `deploy` 目录下已放置 `Dockerfile`、`docker-compose.yml`、`deploy.env`、`start.sh`、`rollback.sh`。

## 首次准备

进入部署根目录并准备目录：

```bash
mkdir -p deploy
cd deploy
```

将联网部署文件放入 `deploy` 目录后，拉取业务代码到同级目录：

```bash
git clone <代码仓库地址> ../java-template
```

授予脚本执行权限：

```bash
chmod +x start.sh rollback.sh
```

按需修改生产部署配置：

```bash
vim deploy.env
```

## 部署配置

`start.sh` 和 `rollback.sh` 默认读取同目录下的 `deploy.env`。如未提供配置文件，则使用脚本内置默认值。

| 配置 | 默认值 | 说明 |
|------|--------|------|
| `ENV` | `prod` | Spring Profile 和镜像 tag 环境前缀 |
| `BRANCH_NAME` | `main` | 部署分支，仅 `start.sh` 使用 |
| `REPO_NAME` | `java-template` | 业务代码目录名和镜像仓库名 |
| `APP_PORT` | `18080` | 宿主机端口，映射到容器 `8080` |
| `APP_LOG_DIR` | `../logs` | 宿主机日志目录，相对 `java-template` 目录解析 |
| `APP_MEMORY_LIMIT` | `1G` | 容器内存上限 |
| `APP_MEMORY_RESERVATION` | `512M` | 容器内存预留 |
| `IMAGE_KEEP_COUNT` | `3` | 每次发布后保留的生产镜像数量 |

脚本会额外导出以下 Compose 变量：

| 变量 | 说明 |
|------|------|
| `APP_IMAGE` | 当前部署或回滚使用的镜像 |
| `APP_CONTAINER_NAME` | 容器名，默认 `${REPO_NAME}-${ENV}` |
| `COMPOSE_PROJECT_NAME` | Compose 项目名，默认 `${REPO_NAME}-${ENV}` |
| `SPRING_PROFILES_ACTIVE` | Spring Profile，默认 `ENV` |
| `APP_BUILD_CONTEXT` | Docker 构建上下文，指向业务代码目录 |
| `APP_DOCKERFILE` | Dockerfile 路径，指向 `deploy/Dockerfile` |

临时覆盖配置示例：

```bash
APP_PORT=18081 APP_LOG_DIR=/data/java-template/logs bash start.sh
```

使用其他配置文件：

```bash
ENV_FILE=/data/deploy/deploy.env bash start.sh
ENV_FILE=/data/deploy/deploy.env bash rollback.sh java-template:prod-20260605170000-abcdef0
```

## 发布

在 `deploy` 目录执行：

```bash
bash start.sh
```

脚本执行流程：

1. 进入 `../java-template` 业务代码目录。
2. 拉取远程分支，并将本地代码重置到 `origin/${BRANCH_NAME}`。
3. 生成镜像版本，例如 `java-template:prod-20260605170000-abcdef0`。
4. 创建日志目录并在 root 执行时调整目录属主。
5. 使用 `deploy/docker-compose.yml` 和 `deploy/Dockerfile` 构建并启动容器。
6. 保留最近 `IMAGE_KEEP_COUNT` 个 `${REPO_NAME}:${ENV}-*` 镜像，并清理悬空镜像。

## 发布后检查

```bash
docker ps --filter name=java-template-prod
docker logs -f java-template-prod
curl -I http://127.0.0.1:18080
```

查看本机保留的生产镜像：

```bash
docker image ls java-template --format "{{.Repository}}:{{.Tag}} {{.CreatedSince}} {{.Size}}" | grep ":prod-"
```

## 回滚

先查看可用历史镜像：

```bash
docker image ls java-template --format "{{.Repository}}:{{.Tag}} {{.CreatedSince}} {{.Size}}" | grep ":prod-"
```

在 `deploy` 目录指定镜像回滚：

```bash
bash rollback.sh java-template:prod-20260605170000-abcdef0
```

回滚脚本执行流程：

1. 校验目标镜像是否存在。
2. 进入 `../java-template` 业务代码目录。
3. 读取 `deploy.env` 并导出 Compose 环境变量。
4. 使用 `deploy/docker-compose.yml` 直接以目标镜像重建并启动 `app-server`。
5. 检查容器是否处于 `running` 状态。

未指定镜像时，脚本会列出本机可用的 `${REPO_NAME}:${ENV}-*` 镜像：

```bash
bash rollback.sh
```

## 注意事项

- 回滚只能使用本机仍然存在的镜像；默认每次发布后只保留最近 3 个生产镜像。
- 回滚只切换应用镜像，不回滚数据库结构和数据。涉及数据库变更时，应提前准备备份和兼容方案。
- `application-prod.yml`、数据库密码、OSS 密钥等敏感配置不要提交到公开仓库。
- 生产环境慎用 `docker compose down -v`，该命令会删除相关数据卷。
