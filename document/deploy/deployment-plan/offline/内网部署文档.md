# 内网部署文档

本文档说明服务器无法访问公网时，如何只依赖部署文件和已构建 jar 部署 `jackal-java-template`。

结论：当前应用部署不需要在服务器上放置项目代码目录。将 `document/deploy/deployment-plan/offline/` 下的部署文件复制到服务器某个目录，再把 jar 放到 `deploy.sh` 同目录，即可构建镜像并启动应用容器。

## 部署内容

内网服务器只需要以下内容：

```text
部署目录/
├── Dockerfile
├── docker-compose.yml
├── deploy.env
├── deploy.sh
├── rollback.sh
└── jackal-java-template-1.0.0-SNAPSHOT.jar
```

其中 jar 文件名不限。部署目录下只有一个 jar 时，`deploy.sh` 会自动选择；有多个 jar 时，需要显式指定 jar 路径。

## 前置条件

内网服务器需提前安装：

- Docker Engine
- Docker Compose Plugin
- `tar` / `gzip`

应用运行时基础镜像必须提前存在于内网服务器，默认是：

```text
eclipse-temurin:21-jre
```

可在可访问镜像仓库的机器上准备 tar：

```bash
docker pull eclipse-temurin:21-jre
docker save -o eclipse-temurin-21-jre.tar eclipse-temurin:21-jre
```

上传到内网服务器后加载：

```bash
docker load -i eclipse-temurin-21-jre.tar
docker image inspect eclipse-temurin:21-jre
```

如果 MySQL、Nginx 也需要在内网服务器上容器化部署，应另外准备并加载对应镜像 tar，例如 `mysql:8.4`、`nginx:1.30.2`。应用部署脚本本身只校验 `RUNTIME_IMAGE`。

## 准备目录

示例目录：

```bash
sudo mkdir -p /data/java-template/deploy /data/java-template/logs
sudo chown -R $USER:$USER /data/java-template
cd /data/java-template
```

将 `document/deploy/deployment-plan/offline/` 下的文件复制到：

```text
/data/java-template/deploy/
```

将外部构建好的 jar 上传到同一目录：

```text
/data/java-template/deploy/jackal-java-template-1.0.0-SNAPSHOT.jar
```

## 构建 jar

在可联网开发机或构建机执行：

```bash
mvn -q -DskipTests package
```

构建产物默认位于：

```text
target/jackal-java-template-1.0.0-SNAPSHOT.jar
```

内网服务器不需要 Maven 依赖，也不需要项目源码。

## 部署配置

`deploy.sh` 和 `rollback.sh` 默认读取同目录下的 `deploy.env`。常用配置如下：

```env
ENV=prod
REPO_NAME=java-template
APP_PORT=18080
APP_LOG_DIR=../logs
APP_MEMORY_LIMIT=1G
APP_MEMORY_RESERVATION=512M
IMAGE_KEEP_COUNT=3
RUNTIME_IMAGE=eclipse-temurin:21-jre
JAR_DIR=.
```

路径配置规则：

- 绝对路径按原值使用。
- 相对路径按 `deploy.sh` 所在目录解析。
- `JAR_DIR=.` 表示 jar 默认放在 `deploy.sh` 同目录。

## 启动应用

进入部署根目录后执行：

```bash
bash deploy/deploy.sh
```

如果部署目录下有多个 jar，显式指定本次发布的 jar：

```bash
bash deploy/deploy.sh /data/java-template/deploy/jackal-java-template-1.0.0-SNAPSHOT.jar
```

脚本会依次执行：

1. 读取 `deploy.env`。
2. 查找或校验 jar 文件。
3. 校验 `RUNTIME_IMAGE` 是否已加载。
4. 使用部署目录下的 `Dockerfile` 构建应用镜像。
5. 写入 `app-image.env`，记录本次镜像和 jar。
6. 使用 `docker-compose.yml` 启动或替换应用容器。
7. 保留最近 `IMAGE_KEEP_COUNT` 个应用镜像，并清理悬空镜像。

镜像 tag 示例：

```text
java-template:prod-20260608120000-jackal-java-template-1.0.0-SNAPSHOT
```

## 发布后检查

查看容器：

```bash
docker ps
```

查看应用日志：

```bash
docker logs --tail=200 java-template-prod
```

检查应用端口：

```bash
curl -I http://127.0.0.1:18080
```

## 后续发布

每次发布新版本：

1. 在外部环境构建新 jar。
2. 上传新 jar 到部署目录。
3. 执行 `bash deploy/deploy.sh` 或显式指定 jar 路径。

示例：

```bash
bash deploy/deploy.sh /data/java-template/deploy/jackal-java-template-1.0.0-SNAPSHOT.jar
```

后续应用发布不需要重复加载 JRE 镜像，除非 `RUNTIME_IMAGE` 版本发生变化。

## 回滚

查看本机已有应用镜像：

```bash
docker image ls java-template --format "{{.Repository}}:{{.Tag}} {{.CreatedSince}} {{.Size}}" | grep ":prod-"
```

回滚到指定镜像：

```bash
bash deploy/rollback.sh java-template:prod-20260608120000-jackal-java-template-1.0.0-SNAPSHOT
```

回滚只切换应用镜像，不回滚数据库结构和数据。涉及数据库变更的版本上线前，需要提前准备数据库备份和兼容方案。

`deploy.sh` 默认只保留最近 3 个应用镜像。如果需要更长回滚窗口，请调大 `IMAGE_KEEP_COUNT`。

## MySQL 和 Nginx

本内网应用部署流程不依赖项目代码目录，也不负责部署 MySQL 或 Nginx。

如果服务器上已经有可用数据库和入口网关，只需确保 `application-prod.yml` 中的连接地址、账号、密码等配置适配目标环境。

如果需要使用本仓库提供的 MySQL、Nginx 参考配置，可以单独复制以下目录到服务器，并按对应文档部署：

```text
document/deploy/mysql/
document/deploy/nginx/
document/sql/production/
```

这些目录属于中间件和初始化 SQL 的参考资料，不是应用 jar 离线部署的必需项。

## 常见问题

### deploy.sh 提示基础镜像不存在

先加载 JRE 基础镜像：

```bash
docker load -i eclipse-temurin-21-jre.tar
```

并确认 tag 与 `deploy.env` 中一致：

```bash
docker image inspect eclipse-temurin:21-jre
```

### docker compose 启动时仍尝试构建

确认启动命令使用的是：

```bash
docker compose -f deploy/docker-compose.yml up -d --no-build app-server
```

并确认 `APP_IMAGE` 指向已经构建好的应用镜像。

### 应用连不上 MySQL

检查：

- MySQL 服务是否已启动。
- 应用生产配置中的数据库地址是否能从应用容器访问。
- 数据库名、账号、密码是否与目标环境一致。
- 日志中不要输出数据库密码等敏感信息。
