# 联网部署文档

本文档说明生产环境可访问公网时，如何拉取代码、构建镜像、启动服务，以及启动失败时如何回滚到指定历史镜像。

## 目录说明

```text
联网部署/
├── start-prod.sh      # 联网生产发布脚本
├── rollback-prod.sh   # 指定镜像回滚脚本
└── java-template/     # 业务代码仓库，脚本默认从同级目录进入该仓库
```

`start-prod.sh` 和 `rollback-prod.sh` 默认要求业务仓库目录位于脚本同级目录下：

```text
document/deploy/部署方案/联网部署/java-template
```

如服务器实际目录不同，请同步调整脚本中的 `REPO_NAME` 或 `REPO_PATH`。

## 前置条件

- 服务器已安装 Git、Docker、Docker Compose Plugin。
- 服务器可以访问代码仓库和基础镜像仓库。
- 当前用户可以执行 Docker 命令；如未加入 `docker` 组，请使用 `sudo bash` 执行脚本。
- 生产配置 `src/main/resources/application-prod.yml` 已完成数据库、OSS、Redis 等真实环境配置。
- 根目录存在 `Dockerfile` 和 `docker-compose.yml`。

## 首次准备

进入联网部署目录：

```bash
cd document/deploy/部署方案/联网部署
```

拉取业务代码仓库：

```bash
git clone <代码仓库地址> java-template
```

检查脚本权限：

```bash
chmod +x start-prod.sh rollback-prod.sh
```

## 部署配置

`start-prod.sh` 默认配置如下：

| 配置 | 默认值 | 说明 |
|------|--------|------|
| `ENV` | `prod` | Spring Profile 和镜像 tag 环境前缀 |
| `BRANCH_NAME` | `main` | 部署分支 |
| `REPO_NAME` | `java-template` | 业务代码目录和镜像仓库名 |
| `APP_PORT` | `18080` | 宿主机暴露端口，映射到容器 `8080` |
| `APP_LOG_DIR` | `../logs` | 宿主机日志挂载目录 |
| `APP_MEMORY_LIMIT` | `1G` | 容器内存上限 |
| `APP_MEMORY_RESERVATION` | `512M` | 容器内存预留 |
| `IMAGE_KEEP_COUNT` | `3` | 保留最近生产镜像数量 |

`docker-compose.yml` 通过脚本导出的变量启动应用：

| 变量 | 说明 |
|------|------|
| `APP_IMAGE` | 当前部署或回滚使用的镜像 |
| `APP_CONTAINER_NAME` | 容器名，默认 `java-template-prod` |
| `COMPOSE_PROJECT_NAME` | Compose 项目名，默认 `java-template-prod` |
| `SPRING_PROFILES_ACTIVE` | Spring Profile，默认 `prod` |
| `APP_PORT` | 宿主机端口，默认 `18080` |
| `APP_LOG_DIR` | 日志挂载目录，默认 `../logs` |
| `APP_MEMORY_LIMIT` | 容器内存上限，默认 `1G` |
| `APP_MEMORY_RESERVATION` | 容器内存预留，默认 `512M` |

如需临时覆盖配置，可在执行脚本前设置环境变量：

```bash
APP_PORT=18081 APP_LOG_DIR=/data/java-template/logs bash start-prod.sh
```

如需调整容器资源限制：

```bash
APP_MEMORY_LIMIT=2G APP_MEMORY_RESERVATION=1G bash start-prod.sh
```

## 发布步骤

在联网部署目录执行：

```bash
bash start-prod.sh
```

脚本会执行以下操作：

1. 进入 `java-template` 仓库目录。
2. 拉取远程 `main` 分支，并将本地代码重置到 `origin/main`。
3. 根据环境、时间和 Git 提交号生成镜像，例如 `java-template:prod-20260605170000-abcdef0`。
4. 创建日志目录并按当前运行用户调整目录属主。
5. 执行 `docker compose -f docker-compose.yml up -d --build` 构建并启动服务。
6. 保留最近 `IMAGE_KEEP_COUNT` 个 `java-template:prod-*` 镜像，并清理悬空镜像。

## 发布后检查

查看容器状态：

```bash
docker ps --filter name=java-template-prod
```

查看启动日志：

```bash
docker logs -f java-template-prod
```

确认端口监听：

```bash
curl -I http://127.0.0.1:18080
```

查看当前保留的生产镜像：

```bash
docker image ls java-template --format "{{.Repository}}:{{.Tag}} {{.CreatedSince}} {{.Size}}" | grep ":prod-"
```

## 回滚步骤

如果新版本启动失败，先查看可用历史镜像：

```bash
docker image ls java-template --format "{{.Repository}}:{{.Tag}} {{.CreatedSince}} {{.Size}}" | grep ":prod-"
```

选择需要回滚的镜像后执行：

```bash
bash rollback-prod.sh java-template:prod-20260605170000-abcdef0
```

回滚脚本会执行以下操作：

1. 校验目标镜像是否存在。
2. 进入 `java-template` 仓库目录。
3. 导出与生产部署一致的 Compose 环境变量。
4. 执行 `docker compose -f docker-compose.yml up -d --no-build app-server`，直接使用目标镜像重建并启动容器。
5. 检查容器是否处于 `running` 状态。

回滚完成后再次检查日志和端口：

```bash
docker logs --tail=200 java-template-prod
curl -I http://127.0.0.1:18080
```

如果未指定镜像，脚本会列出本机可用的 `java-template:prod-*` 镜像：

```bash
bash rollback-prod.sh
```

## 注意事项

- 回滚只能使用本机仍然存在的镜像；`start-prod.sh` 默认只保留最近 3 个生产镜像。
- 如果需要更长回滚窗口，请调大 `IMAGE_KEEP_COUNT`，或将镜像推送到私有镜像仓库。
- 回滚只切换应用镜像，不回滚数据库结构和数据。涉及数据库变更的发布，需要提前准备数据库备份和兼容方案。
- `application-prod.yml`、数据库密码、OSS 密钥等敏感配置不要提交到公开仓库。
- 执行 `docker compose down -v` 会删除相关数据卷，生产环境慎用。
